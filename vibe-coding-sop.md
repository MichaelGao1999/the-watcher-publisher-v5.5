# Vibe Coding 五阶段工作流（SOP）

> 本文档是从 [项目名] 项目实践中提取的标准操作流程，可复用于其他 AI 驱动的软件开发项目。
> 核心理念：**人负责决策，AI 负责执行，Orchestrator 负责接力。**

---

## 项目起点判断

根据项目当前状态选择切入点，不要机械地从阶段一开始：

| 项目状态 | 切入阶段 | 动作 |
|---------|---------|------|
| **纯新项目**（只有想法，无代码无文档） | 阶段一 | 从需求讨论开始 |
| **有需求文档**（已有 PRD/需求说明） | 阶段二 | 直接生成设计文档 |
| **有设计文档**（已有架构/接口设计） | 阶段三 | 直接拆分子任务 |
| **有 Prompt**（已有 orchestrator 指令） | 阶段五 | 直接执行开发 |
| **半成品项目**（已有代码，文档不全） | 阶段五 | 先评估现有代码，再决定回溯补文档还是继续开发 |

**半成品项目特别注意**：
1. Agent 先读取全部现有代码，输出《代码评估报告》：架构现状、已完成模块、缺失模块、技术债务
2. 根据评估结果决策：
   - 代码结构清晰 → 继续阶段五开发
   - 架构混乱/缺设计 → 回溯阶段二，先补 `docs/design.md`，再重新进入阶段五
   - 模块边界不清 → 回溯阶段三，先拆任务再开发

---

## 阶段一：讨论需求

**目标**：把模糊的想法变成清晰的需求文档

**输入**：用户的原始想法、业务场景、目标用户画像

**输出**：`docs/proposal.md`

**AI 行为规范**：
- 用提问方式帮助用户确认需求
- **绝不猜测用户意图**
- 所有不明确的地方必须提问
- 用户没有编程背景时，用大白话解释技术概念

**关键产出**：
- 目标确认
- 输入/输出清单（用户能做什么、系统展示什么）
- 步骤/流程确认
- 待确认清单（打勾后才能继续）

**技术需求确认（阶段二的输入条件）**：

以下问题必须在阶段一结束时得到明确答案，直接影响阶段二需要产出哪些文件：

| # | 问题 | 选项 | 影响 |
|---|------|------|------|
| T-01 | 是否需要数据持久化存储？ | ① 不需要 ② 本地文件（JSON/YAML/SQLite） ③ 远程数据库 | ②/③ → 阶段二需要产出 `docs/database.md` |
| T-02 | 是否需要对外提供 API / 服务接口？ | ① 不需要 ② 仅内部模块调用 ③ 对外暴露 REST/GraphQL/gRPC | ②/③ → 阶段二 `design.md` 中需要包含接口契约章节 |
| T-03 | 是否需要独立部署？ | ① 纯本地运行 ② 需要部署到服务器/云平台 ③ 需要容器化 | ②/③ → 阶段四需要产出 `deploy.md` |
| T-04 | 是否有用户认证/权限体系？ | ① 不需要 ② 简单本地认证 ③ 需要完整的用户系统 | ②/③ → 阶段二 `design.md` 中需要包含安全设计章节 |

> **为什么放在阶段一？** 这些问题的答案会影响阶段二需要产出哪些文件，以及 design.md 中需要包含哪些章节。阶段二开始后再反问"要不要数据库"会打断设计节奏。

---

## 阶段二：设计文档搭建

**目标**：根据需求文档生成概要设计文档

**输入**：`docs/proposal.md`

**输出**：
- `docs/design.md`（技术设计文档：架构总览、模块划分、接口契约、数据流）
- `docs/brief.md`（已确认决策摘要，大白话版，给把控者看）
  → **性质：阶段三~五的不可变更的决策边界**。实现与此矛盾时，直接引用纠偏，不复议

**AI 行为规范**：
- 根据需求划分模块（每个模块只干一件事）
- 模块之间尽量独立，可单独测试
- 定义模块间接口（函数签名）
- **接口设计要前置**：模块间如何调用、数据如何流转，先约定再实现，禁止边写代码边补接口
- 不猜测意图，不明确的地方提问
- 技术选型需经用户确认，AI 不得擅自决定框架/语言变更
- **每确认一项决策，同步用大白话记录到 docs/brief.md**：用户没有编程背景，docs/design.md 里的技术术语用户可能看不懂，brief.md 用一行一句大白话写清楚"我们定了什么、为什么定、不能改什么"
- **本阶段交付完成后即结束**：不要建议用户继续下一阶段，用户会自行开启新会话进入阶段三

**关键产出**：
- 模块划分图（文字版即可）
- 数据流图：不仅画模块盒子，还要画核心数据实体如何进出各模块
- 每个模块的职责 + 对外接口
- 模块间调用关系矩阵
- 数据持久化方案
- 测试策略
- 接口签名（函数/事件/数据结构）
- `docs/brief.md`：已确认决策的大白话摘要（编号 + 决策内容 + 来源），用于阶段三~五的纠偏和验收

---

## 阶段三：划分任务

**目标**：为每个模块拆分成最小可执行子任务

**输入**：
- `docs/proposal.md`
- `docs/design.md`
- `docs/brief.md`（阶段二已确认的决策摘要，避免拆分任务时违背已定边界）

**输出**：
- `docs/tasks/task-{module}.md`（每个模块一个文件）
- `docs/tasks/task-progress.md`（总进度看板）

**AI 行为规范**：
- 每个子任务用 `[ ]` checkbox 表示
- 子任务要小到"一个 subAgent 一次能做完"
- 包含测试点（每个模块必须测什么）
- 标注优先级（P1 必须做 / P2 后续迭代）
- **如范围或优先级发生调整，同步更新 docs/brief.md**

**关键产出**：
- 每个模块 10~50 个子任务
- 子任务含编号（如 `B-01`, `B-02`）
- 依赖关系与推荐开发顺序
- 总进度表（模块状态 + 子任务进度）

- **本阶段交付完成后即结束**：不要建议用户继续下一阶段，用户会自行开启新会话进入阶段四

---

## 阶段四：生成 SubAgent Prompt

**目标**：生成 vibe coding 用的主控 prompt

**输入**：
- `docs/proposal.md`
- `docs/design.md`
- `docs/brief.md`
- `docs/tasks/*.md`

**输出**：`prompt.md`（项目根目录）

**AI 行为规范**：
- 阅读所有输入，理解工程全貌
- 评估现有代码（如果有）
- 不明确的地方必须向用户提问
- prompt 要让主 Agent 和 subAgent 都知道该干嘛
- **本阶段交付完成后即结束**：不要建议用户继续下一阶段，用户会自行开启新会话进入阶段五

**Prompt 必须包含的内容**：
1. **项目架构**：模块列表、依赖顺序、接口规范
2. **现有代码评估**：哪些保留、哪些重构、哪些新建
3. **主 Agent 职责**：调度逻辑、验收标准、派发模板
4. **SubAgent 职责**：开发流程、代码规范、测试要求
5. **测试规范**：测试方法、覆盖率要求、验收 checklist
6. **依赖关系与数据契约**：调用矩阵、数据结构设计
7. **已知确认事项**：设计文档中已确认决策，避免重复提问（来源：docs/brief.md）
8. **目标目录结构**：最终文件树
9. **验收标准**：DoD checklist（含 docs/brief.md 逐条核对）
10. **决策摘要引用**：prompt 中注明"如输出与 docs/brief.md 矛盾，Orchestrator 直接引用纠偏"

**关键产出**：
- 一个完整的 prompt.md
- 新会话只需读取 prompt.md 即可开始执行

---

## 阶段五：执行开发（SubAgent 接力）

**目标**：按批次逐个实现模块，每轮控制在 subAgent 超时范围内

**输入**：
- `prompt.md`（主指令，含架构和接口规范）
- `docs/tasks/task-progress.md`（当前完成进度）
- `docs/design.md`（模块详细设计）
- `docs/tasks/task-{module}.md`（子任务清单）

**输出**：各模块源码 + 测试 + 集成

**AI 行为规范**：
- **开发前先读项目规则文件 + docs/brief.md**：了解项目规则与不可违反的已确认决策
- **不要一把梭**：一个 Agent 15 分钟写不完完整模块（代码+测试+调通）
- **拆成多轮接力**：每轮有明确交付物，超时也不怕，下轮接着干
- **Orchestrator 先读后派**：每轮开始前检查"前人留下了什么"，再生成精确接力指令
- **争议时引用 brief.md 纠偏**：如果 subAgent 输出与已定决策矛盾，Orchestrator 直接引用 docs/brief.md 第 X 条，不需要重新解释理由

**接力轮次模板**（参考）：

| 轮次 | 任务 | 交付物 | 预估时间 |
|------|------|--------|---------|
| 第1轮 | 写核心代码框架 | 接口 + 基础功能实现 | 10~15 分钟 |
| 第2轮 | 写测试+调通 | 单元测试 + 集成验证 | 10~15 分钟 |
| 第3轮 | 补 UI/集成 | 前端页面 / 样式 / 联调 | 5~10 分钟 |

**Orchestrator 接力流程**：

```
Step 1: 检查"前人留下了什么"
        → 读目标文件，看哪些接口已实现、哪些为空
        → 读测试文件，看测试覆盖率

Step 2: 生成精确的接力指令
        → 明确告诉新 Agent：已完成什么、缺失什么、不要改什么

Step 3: 派发新 Agent，验收产出
        → 只验收本轮交付物，不跨轮追责

Step 4: 如果又超时，重复 Step 1~3
        → 每次接力都基于最新代码状态
```

**关键原则**：

| 原则 | 说明 |
|------|------|
| **先读后做** | 接力 Agent 必须先读取现有代码，了解当前状态再动手 |
| **只补不推翻** | 明确约束"不要重写已有代码"，避免返工 |
| **交付物要小** | 一轮只交付一个文件（或一个函数），确保 15 分钟内能完成 |
| **状态透明** | 每轮结束后 Orchestrator 要记录"完成了什么、还剩什么" |
| **会话边界** | 每轮接力由用户开启新会话触发，Agent 不主动连续执行多轮 |

## 可选产出与触发条件

以下文件不是每个项目都需要，由阶段一的技术需求确认（T-01 ~ T-04）决定是否在对应阶段产出：

| 阶段 | 可选产出 | 触发条件 | 说明 |
|------|---------|---------|------|
| 阶段二 | `docs/database.md` | T-01 答案为 ② 或 ③ | 数据库/持久化方案设计文档 |
| 阶段二 | `design.md` 中的接口契约章节 | T-02 答案为 ② 或 ③ | 模块间或对外接口的函数签名、数据结构 |
| 阶段二 | `design.md` 中的安全设计章节 | T-04 答案为 ② 或 ③ | 认证、授权、数据安全策略 |
| 阶段四 | `deploy.md` | T-03 答案为 ② 或 ③ | 部署架构、环境配置、CI/CD 流程 |

**判断原则**：
- 触发条件满足 → **必须**产出对应文件，不得省略
- 触发条件不满足 → **不得**产出对应文件，避免过度设计
- 判断依据以 `docs/proposal.md` 中的技术需求确认为准，Agent 不得擅自猜测

## 附：跨阶段通用规则

| 规则 ID | 规则内容 |
|---------|---------|
| R-01 | 需求不明 → 停止，提问，不猜测 |
| R-02 | 决策已入 brief.md → 后续阶段只引用，不复议 |
| R-03 | 子任务必须可测试 → 无测试点的任务不合格 |
| R-04 | 一个 subAgent 一轮只做一件事 → 超时可控，接力可行 |
| R-05 | Orchestrator 是代码状态的守护者 → 每轮先读后派 |
| R-06 | 阶段边界 = 会话边界 → 每个阶段完成后结束当前会话，下阶段由用户开启新会话 |

## 文件树约定

```
[项目名]/
├── AGENTS.md                 # 基础设施：项目硬规则（启动时建立）
├── vibe-coding-sop.md        # 基础设施：五阶段工作流参考（启动时建立）
├── status.md                 # 基础设施：当前进度/待办（启动时建立）
├── session-log.md            # 基础设施：会话历史（启动时建立，后续追加）
├── decisions.md              # 基础设施：关键决策记录（启动时建立，后续追加）
├── troubleshooting.md        # 基础设施：问题索引（启动时建立，后续追加）
├── lessons-learned.md        # 基础设施：跨项目经验（启动时建立，后续追加）
├── docs/
│   ├── proposal.md           # 阶段一产出
│   ├── design.md             # 阶段二产出
│   ├── brief.md              # 阶段二产出（决策摘要）
│   ├── tasks/
│   │   ├── task-{module}.md  # 阶段三产出
│   │   └── task-progress.md  # 阶段三产出（总进度）
├── prompt.md                 # 阶段四产出
└── src/                      # 阶段五产出
```

> **分层说明**：
> - **基础设施**（`AGENTS.md`、`status.md`、`session-log.md` 等）：项目启动时立刻建立，与所处阶段无关。阶段一确认需求时就会有关键决策，需要立即记录。
> - **阶段产出**（`docs/*`、`prompt.md`、`src/`）：按 SOP 五阶段逐步生成，根据项目状态选择切入点。

## 附录 B：跨项目知识同步 SOP

> 当用户说「同步知识」「拉取经验」「聚合知识」时执行本流程。
> 目标：将用户 GitHub 所有项目中的 `decisions.md`、`lessons-learned.md`、`troubleshooting.md` 聚合到本项目，形成跨项目知识母库。

### 配置初始化（首次执行）

1. 检查 `config/github-sync.json` 是否存在
   - 不存在 → 从骨架模板复制，提示用户填写 `username` 和 `token`
   - 存在但 `username` 为空 → 提示用户填写
2. 确认 `includeRepos` 和 `excludeRepos`：
   - 两者都为空 → 同步该用户的**全部**仓库
   - `includeRepos` 非空 → 只同步列表中的仓库
   - `excludeRepos` 非空 → 排除列表中的仓库

### 执行方式 A：自动化脚本（推荐）

**前提**：环境已安装 Python 和 `requests`

```bash
# 1. 安装依赖（如未安装）
pip install requests

# 2. 运行同步脚本
python scripts/sync-knowledge.py
```

脚本会自动完成：
- 调用 GitHub API 获取仓库列表
- 拉取每个仓库的目标文件
- 解析、去重、标注来源
- 合并到本项目的母库文件
- 输出同步报告

### 执行方式 B：AI 原生手动流程（Fallback）

当脚本环境不可用时，AI 按以下步骤手动执行：

**Step 1 — 读取配置**
- 读取 `config/github-sync.json` 获取 `username`、`token`、`includeRepos`、`excludeRepos`

**Step 2 — 获取仓库列表**
- 使用 `FetchURL` 调用 `https://api.github.com/users/{username}/repos?per_page=100`
- 如配置了 Token，在 Header 中附加 `Authorization: Bearer {token}`
- 按 `includeRepos`/`excludeRepos` 过滤

**Step 3 — 拉取知识文件**
- 对每个仓库，尝试拉取以下 URL：
  - `https://raw.githubusercontent.com/{username}/{repo}/{branch}/decisions.md`
  - `https://raw.githubusercontent.com/{username}/{repo}/{branch}/lessons-learned.md`
  - `https://raw.githubusercontent.com/{username}/{repo}/{branch}/troubleshooting.md`
- 404 则跳过（该仓库无此文件）

**Step 4 — 备份母库**
- 将本项目的 `decisions.md`、`lessons-learned.md`、`troubleshooting.md` 复制到 `.backup/` 目录，命名带时间戳

**Step 5 — 解析与去重**

| 文件 | 解析方式 | 去重规则 |
|------|---------|---------|
| `decisions.md` | 提取 `## ADR-xxx:` 区块 | **不去重**，全部追加 |
| `lessons-learned.md` | 提取表格行和列表项 | 按「经验描述」去重，合并来源标签 |
| `troubleshooting.md` | 提取 `### [错误关键词]` 区块 | 按「错误关键词」去重，保留最全解决方案 |

**Step 6 — 标注来源**
- 所有并入的内容追加来源标签：`[来源:仓库名 @YYYY-MM-DD]`
- 本项目原有内容标记为 `[母库]`

**Step 7 — 合并写入**
- 使用 `WriteFile`/`StrReplaceFile` 将合并后的内容写回母库文件
- 保持原有 Markdown 结构，新内容追加到对应章节末尾

**Step 8 — 输出报告**

```markdown
【知识同步报告】
📦 同步仓库数：X
📄 拉取文件数：Y
✨ 新增条目数：Z

📁 更新文件：
  - decisions.md：+A 条
  - lessons-learned.md：+B 条
  - troubleshooting.md：+C 条

💾 备份位置：.backup/
```

**Step 9 — 更新状态**
- 追加 `status.md`「更新记录」：记录本次同步时间和摘要

### 合并规范（不可违反）

1. **来源必须标注**：每条并入内容必须有 `[来源:仓库名 @日期]`
2. **合并前必须备份**：备份到 `.backup/`，命名带时间戳
3. **去重策略**：
   - `lessons-learned` → 按描述去重，相同经验合并来源
   - `troubleshooting` → 按关键词去重，不同场景的解决方案并列保留
   - `decisions` → 不去重，ADR 上下文差异可能很大
4. **母库优先**：本项目原有内容优先保留，新内容追加
5. **空文件跳过**：源文件为空或为模板占位符时跳过

### 验证清单

同步完成后，AI 应自检以下项目：

- [ ] `config/github-sync.json` 配置正确且已保存
- [ ] `.backup/` 目录存在且包含本次备份
- [ ] 所有新增条目都标注了 `[来源:仓库名 @日期]`
- [ ] 无重复条目（按去重规则检查）
- [ ] `status.md` 已更新同步记录
- [ ] 同步报告已输出给用户
